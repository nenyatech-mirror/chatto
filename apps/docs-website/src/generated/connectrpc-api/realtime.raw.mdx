Chatto exposes these ConnectRPC services below `/api/connect`.

Endpoint paths use the Connect convention:

`/api/connect/<fully-qualified-service>/<method>`

Use this reference when building clients against Chatto's public ConnectRPC API.


## Supporting Types

<a id="chatto-realtime-v1-RealtimeCaughtUp"></a>

### RealtimeCaughtUp

Confirms that durable replay is complete and live delivery has begun.

Clients can retain `cursor` after applying every preceding event. A cursor
must not outlive the projection state it describes. Events after this frame
are live. Transient events are never replayed. A cursor is usable for up to
24 hours after issue and should be replaced by each later caught-up cursor.

| Field | Type | Description |
| --- | --- | --- |
| `cursor` | `string` | Opaque cursor at the replay-to-live handoff boundary. |



<a id="chatto-realtime-v1-RealtimeClientFrame"></a>

### RealtimeClientFrame

Client-to-server frame for Chatto's protobuf WebSocket realtime protocol.

Clients send binary protobuf frames to `/api/realtime`. The first frame must
be `hello`; after the server replies with `hello`, clients send
`subscribe_events` to start the authenticated server-projection stream.

| Field | Type | Description |
| --- | --- | --- |
| `frame.hello` | [`RealtimeClientHello`](#chatto-realtime-v1-RealtimeClientHello) | Opens the protocol session and optionally carries a bearer token. |
| `frame.subscribe_events` | [`RealtimeSubscribeEvents`](#chatto-realtime-v1-RealtimeSubscribeEvents) | Starts the caller's authorized server-projection stream. |
| `frame.ping` | [`RealtimePing`](#chatto-realtime-v1-RealtimePing) | Application-level ping. The server replies with `pong`. |
| `frame.hydrate_room` | [`RealtimeHydrateRoom`](#chatto-realtime-v1-RealtimeHydrateRoom) | Adds one joined room's recent timeline to the retained projection. |

<a id="chatto-realtime-v1-RealtimeHydrateRoom"></a>

### RealtimeHydrateRoom

Requests lazy materialisation of one joined room's recent timeline.

The server replies on the same ordered projection stream with a
`room_timeline_replace` operation. Once hydrated, later timeline mutations
for the room are included for the lifetime of this connection. Clients send
all still-retained room IDs again in their next subscription. A connection
may retain at most 64 distinct room IDs; excess requests receive a
non-fatal `too_many_retained_rooms` error.

| Field | Type | Description |
| --- | --- | --- |
| `room_id` | `string` | Joined room to materialise. Repeating a retained room is idempotent. |

<a id="chatto-realtime-v1-RealtimeProjectionEvent"></a>

### RealtimeProjectionEvent

One ordered, idempotent mutation of the caller's authorised server view.

A fresh subscription starts with a reset operation followed by current
resource upserts. A resumed subscription derives operations from EVT after
the supplied cursor. Both paths use this envelope and the same client
reducer. The stream is a convergence feed, not an immutable audit log:
replay uses current authorisation, deletion, and erasure state.

| Field | Type | Description |
| --- | --- | --- |
| `id` | `string` | Stable event ID. Synthetic compacted-replay events use server-generated IDs. |
| `created_at` | `google.protobuf.Timestamp` | Time the source fact occurred or the compacted event was generated. |
| `actor_id` | `optional string` | User or system actor that caused the source fact, when known. |
| `resume_cursor` | `optional string` | Opaque cursor safe to retain with the resulting projection state after applying every operation in this event. |
| `operations` | repeated [`RealtimeProjectionOperation`](#chatto-realtime-v1-RealtimeProjectionOperation) | Operations applied atomically and in order by the client reducer. |
