Chatto can transcode uploaded videos into multiple quality renditions for adaptive HLS playback. When a user uploads a video, the server stores HLS segments and a thumbnail, generates playlists on demand, and serves them through a built-in video player. Processed video audio is normalized to AAC stereo for consistent browser playback, including for uploads with unusual multichannel layouts.

## Prerequisites

Video processing requires **ffmpeg** and **ffprobe** to be installed on the server. The official Chatto Docker image includes both.

If you're running Chatto outside Docker, install ffmpeg through your system's package manager:

```bash
# Debian/Ubuntu
apt install ffmpeg

# macOS (Homebrew)
brew install ffmpeg

# Alpine
apk add ffmpeg

# FreeBSD
pkg install ffmpeg
```

## Enabling Video Processing

Add the following to your `chatto.toml`:

```toml
[video]
enabled = true
```

Or set the environment variable:

```bash
CHATTO_VIDEO_ENABLED=true
```

That's it. Chatto will auto-detect `ffmpeg` and `ffprobe` from your `PATH`.

<Aside>
  When video processing is disabled (the default), Chatto rejects new video uploads in the composer and API. Enabling processing allows video uploads and adds transcoding, adaptive playback, and thumbnail generation.
</Aside>

## Configuration Options

All settings live under the `[video]` section in `chatto.toml`. See the [Environment Variables](/reference/environment-variables/#video-processing) reference for the full list.

| Setting | Default | Description |
|---------|---------|-------------|
| `enabled` | `false` | Enable video processing |
| `ffmpeg_path` | auto-detect | Path to `ffmpeg` binary |
| `ffprobe_path` | auto-detect | Path to `ffprobe` binary |
| `max_concurrent` | `2` | Max concurrent transcoding jobs |
| `max_upload_size` | `100 MB` | Max upload size for video files |
| `temp_dir` | system temp | Directory for temporary transcoding files |

### Tuning Concurrency

Each transcoding job uses significant CPU. The default of 2 concurrent jobs is conservative. On a dedicated server with more cores, you can increase this:

```toml
[video]
enabled = true
max_concurrent = 4
```

### Custom Binary Paths

If `ffmpeg` and `ffprobe` aren't in your `PATH`, specify their locations explicitly:

```toml
[video]
enabled = true
ffmpeg_path = "/usr/local/bin/ffmpeg"
ffprobe_path = "/usr/local/bin/ffprobe"
```

## How It Works

1. A user uploads a video through the chat composer
2. The original file is stored immediately and the message is posted
3. The video appears with a "Processing..." indicator
4. In the background, Chatto creates temporary MP4 renditions, packages them into six-second HLS segments, and generates a thumbnail
5. Once processing completes, Chatto generates authorised playlists on demand and the player uses adaptive HLS playback

If processing fails (e.g. corrupt file, unsupported codec), the user sees an error message and can try uploading again.

## Storage Considerations

Video transcoding persists the original upload, a thumbnail, and HLS segments. The temporary MP4 renditions and generated playlists do not consume durable asset storage. If you expect significant video usage, consider:

- **Using S3 storage** for assets (`core.assets.storage_backend = "s3"`) to avoid filling up NATS JetStream storage
- **Providing a fast temp directory** with enough free space for in-progress transcoding jobs

Processed videos are seekable on both NATS ObjectStore and S3 because HLS uses bounded segments instead of byte ranges into one large object. Videos processed before HLS support are not backfilled; they continue using their existing MP4 variants.

<Aside type="caution">
  During an upgrade to an HLS-capable release, update every Chatto replica behind the same endpoint before allowing new video uploads. Older replicas cannot expose HLS metadata or serve the playlist and segment routes required by newly processed HLS-only videos.
</Aside>
