Chatto's permissions system is built around two ideas: **roles** (which group people) and **permissions** (which describe capabilities). Every authenticated user holds at least one role; every action the user attempts is checked against the permissions their roles grant. You configure both per server.

This page explains the model from the operator's perspective. It does not assume you've used a permissions system before.

## The four built-in roles

Every Chatto server starts with four roles. You can't delete them and you can't change their relative rank.

| Role         | Rank | Default capabilities                                           |
| ------------ | ---- | -------------------------------------------------------------- |
| **everyone** | 0    | Posting, reading, reacting, joining rooms, sending DMs.        |
| **moderator** | 100 | Everyone's permissions plus member removal and message moderation (edit/delete any message). |
| **admin**    | 900  | Every administrative permission individually granted: managing rooms, roles, members, server settings, viewing admin pages. |
| **owner**    | 1000 | Same enumerated permission set as admin. The distinction is purely rank — owners can manage admin-rank users, admins cannot manage owners. |

**Higher rank number means more power.** A moderator (100) outranks everyone (0); admin (900) outranks moderator; owner (1000) sits at the top. The wide gaps between system roles leave room for custom roles to slot in at any rank.

`everyone` is implicit. Every authenticated user automatically has it — you don't assign it manually. The other three are assigned with `chatto reset rbac` (for the initial owner via `owners.emails`) or through the admin UI.

## How a permission check works

When a user tries to do something — post a message, delete a channel, kick a member — Chatto runs a short decision procedure:

1. **DM privacy boundary.** If the action is happening in a DM and falls under the privacy floor (moderating someone else's DM message, browsing the DM list, inviting people), the answer is always **no**, regardless of any role grant.
2. **User-level overrides.** If you've granted or denied this specific permission to this specific user (server-wide or in a specific room), that decision wins. User-level overrides beat every role.
3. **Role hierarchy walk.** Otherwise, Chatto checks the user's roles in rank order, highest first. For each role it looks at the room-specific decision (if any) before the server-wide decision. The first explicit allow or deny it finds is the answer.
4. **Default deny.** If no role has an opinion, the action is denied.

There is no "skip the rules" shortcut. Owners pass permission checks because the owner role is seeded with every server-scope permission explicitly granted — the same way admin gets its permissions. Any deny you configure applies uniformly.

<Aside type="tip">
**Two rules to remember.** Across roles, **higher rank wins**. Within a role, **room-specific decisions win over server-wide ones**. Everything else is detail.
</Aside>

## Permissions

Permissions are named capabilities like `message.post`, `room.manage`, `admin.view-users`. There are around 30 of them grouped into categories (room, message, member, role, admin, DM, user). Each one is either granted or denied to a role (or a specific user) at server scope, or — for room-relevant permissions — at a specific room.

A few worth knowing:

| Permission         | Meaning                                                       |
| ------------------ | ------------------------------------------------------------- |
| `message.post`     | Post new messages in a channel                                |
| `room.list`        | See a room in the room list (per-room → hide channels)        |
| `room.join`        | Join a channel                                                |
| `room.manage`      | Rename, archive, or delete any room                           |
| `message.edit-any` / `message.delete-any` | Moderate other users' messages (rank-aware) |
| `dm.view` / `dm.write` | Access DMs and send direct messages                       |
| `role.manage`      | Create, edit, or delete roles                                 |
| `role.assign`      | Assign roles to users (administer them)                       |
| `user.delete-any`  | Delete any user's account (rank-aware; admin power)           |
| `user.delete-self` | Delete your own account                                       |

The full list is visible in the admin UI under **Roles → Permissions**, with descriptions for each.

## Common patterns

### Read-only announcements channel

Goal: everyone can read `#announcements`, but only admins and moderators can post.

1. Open the room's **Permissions** tab in the admin UI.
2. Deny `message.post` for the **everyone** role at this room.
3. Leave `message.post` granted on **moderator** and **admin** (they have it by default).

Why this works: when a regular user tries to post, the walker checks `everyone` and sees the room-level deny → blocked. When a moderator tries to post, the walker checks `moderator` first (higher rank) and sees its server-wide grant → allowed. The `everyone` deny is never reached for higher-ranked roles.

### Private team channels

Goal: a `#engineering` channel that only people on the engineering team can see and join.

1. Create a custom **Engineering** role.
2. On the `#engineering` room, deny `room.list` and `room.join` for **everyone**.
3. On the same room, grant `room.list` and `room.join` for **Engineering**.
4. Assign the **Engineering** role to your team members.

Non-engineering users won't see the room in their sidebar (it's filtered out of the room list) and couldn't join even if they got the URL. Engineering users see it and join normally.

You can do the same with messaging permissions (`message.post`, `message.react`) for read-only or write-only access patterns.

### Suspending a single user

Goal: one user keeps posting things they shouldn't. You want to stop them from posting without removing them from the server.

1. Open the user's profile in the admin UI.
2. Under **Permission overrides**, deny `message.post` for that user (server scope).

User-level denies are stronger than role grants, so this works regardless of which roles the user has — even if the user is an owner. To restore, remove the override.

This is also how you build limited "timeouts": deny posting, reacting, and DM-sending temporarily, then clear the overrides.

### Giving one user admin powers in one room

Goal: someone on your team needs to moderate `#community-help` without becoming a server moderator everywhere else.

1. Open the user's profile.
2. Under **Permission overrides**, grant `message.delete-any` and `message.edit-any` for that user, scoped to `#community-help`.

User-level grants apply only in the room you scoped them to. No new role required.

### Custom roles

For anything that's a recurring pattern across multiple users, make a custom role:

1. **Roles → New role** in the admin UI.
2. Give it a name and pick its rank by dragging it in the role list (between `everyone` and `moderator` is the typical slot).
3. Grant the permissions it should have.
4. Assign the role to users.

Use custom roles when the pattern is "this group of people can do X". Use user-level overrides when it's "this specific person needs an exception".

## Owner vs admin

Owner and admin hold the **same enumerated permission set**. Anything an admin can do, an owner can do — and vice versa, capability-wise. There's no "skip every check" super-permission attached to owner.

What does separate them is rank, and rank applies to two things:

- **Targeted user actions.** When one user administers another (rename them, change their roles, suspend them with a user-level deny), Chatto requires the actor to *strictly outrank* the target. Owners outrank admins, so admins can't administer owners; admins outrank moderators and regular members, so they can. Two owners are peers — neither can administer the other without one being demoted first via the CLI.
- **Self-lockout prevention.** Owners cannot revoke their own owner role through the API; if you need to remove an owner, another owner has to do it, or the operator does it via `chatto reset rbac`.

If you set a user-level deny on a permission for a specific user, that deny applies to **everyone**, including owners. There's no permission grant that sidesteps user-level overrides. (This is by design — it gives you an off-switch for compromised accounts at any rank.)

## DM privacy

DMs run through the same permission system as channels, but with one hard rule: a fixed set of permissions are **always denied** in DM rooms, regardless of any grant.

| Denied in DMs              | Why                                       |
| -------------------------- | ----------------------------------------- |
| `message.edit-any`         | Privacy — operators can't edit DM content |
| `message.delete-any`       | Privacy — operators can't delete DM content |
| `message.echo`             | DMs can't be echoed to channels           |
| `room.manage`              | DM rooms don't have admin controls        |
| `room.list`, `room.create` | DMs have their own listing and creation API |
| `member.invite`, `member.remove` | DMs have their own participation model |

What this means in practice: even an owner can't read into DMs, edit someone's DM messages, or delete them. Access to DMs is gated by participation (you're either in the conversation or you're not), and the actions you can perform inside a DM are limited to what makes sense between participants — posting, replying, editing your own messages, reacting.

If you need any kind of DM moderation, it has to be designed as a separate auditable feature with explicit user consent. "Log in as admin and look" is not supported.

## Rank vs permission

Some actions target another user — renaming them, kicking them, editing their settings, deleting their messages. For these, having the right permission isn't enough. You also have to *outrank* the target by role hierarchy.

Concretely, to act on another user you need both:

1. The relevant permission (e.g. `role.assign` for identity edits, `message.delete-any` for message moderation).
2. To strictly outrank them. Two admins are peers; one admin can't act on another admin even though they have the permission.

This is what stops a rogue moderator from sabotaging admins or owners. The moderator role has `message.edit-any` and `message.delete-any` by default — but the rank check means they can only use those capabilities against users they outrank. They can moderate regular members; they can't touch admins or owners. The owner outranks the moderator structurally; permissions don't change that.

The same rule applies across the board:

| Action                         | Permission           | Rank-checked? |
| ------------------------------ | -------------------- | ------------- |
| Edit own profile / settings    | (self-action)        | n/a           |
| Edit/delete own message        | `message.edit-own` / `message.delete-own` | n/a |
| Edit/delete another user's message | `message.edit-any` / `message.delete-any` | **yes** |
| Rename another user            | `role.assign`        | **yes**       |
| Assign/revoke roles            | `role.assign`        | **yes**       |
| Suspend a user (user-level deny) | `role.assign`      | **yes**       |

If you ever find yourself needing to "fix" another peer's account, the answer is to demote them first, do the work, then re-promote them — not to grant yourself a permission to override the rank check.

## Where this is configured

| What you want to do                          | Where                              |
| -------------------------------------------- | ---------------------------------- |
| Set the initial owner                        | `owners.emails` in `chatto.toml`   |
| Create or edit roles                         | Admin UI → **Roles**               |
| Assign roles to users                        | Admin UI → **Members**             |
| Edit role permissions                        | Admin UI → **Roles → [role name]** |
| Set room-specific overrides                  | Room admin → **Permissions**       |
| Set per-user overrides                       | Admin UI → **Members → [user]**    |
| Reset everything to defaults                 | `chatto reset rbac` (CLI)          |

`owners.emails` is matched against **verified** emails only — claiming an address you can't receive mail at won't grant you ownership. On a fresh server, the first user whose verified email matches that list is auto-promoted to owner; no restart required.

<Aside type="caution">
  `chatto reset rbac` wipes all roles and permission grants and reseeds the defaults. The seeded owner and admin roles get the full enumerated server-scope permission set (they differ only in rank); moderator gets its moderation set; everyone gets the default member permissions. Custom roles, role assignments, and overrides are deleted.
</Aside>
