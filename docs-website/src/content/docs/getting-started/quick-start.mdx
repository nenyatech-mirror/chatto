## Introduction

Everything you need to get your own Chatto server started is bundled into a single `chatto` binary. Depending on where and how you want to host it,
there's also a Docker image available. For production-style examples, see the deployment guides for Docker Compose, the standalone binary, and high-availability setups.

## Using Docker

<Aside type="caution">
  This Docker command-line setup is recommended only for quickly testing Chatto locally. If you want to deploy Chatto on a server, use the [Docker Compose guide](/guides/dockercompose/) instead.
</Aside>

You're on the Quick Start page, so let's start quickly. The `ghcr.io/chattocorp/chatto` image contains the Chatto server and the embedded web UI. It keeps configuration in `/config/chatto.toml` and embedded NATS data in `/data`, so persist both paths across container restarts.

Create a local directory for the first run so the generated config and data are easy to inspect.

<Steps>

1. **Create a project directory with configuration and data folders:**

   ```bash
   mkdir -p chatto/config chatto/data
   cd chatto
   ```

2. **Run `chatto init` to create the configuration file:**

   ```bash
   docker run --rm \
     -e PUID="$(id -u)" \
     -e PGID="$(id -g)" \
     -v ./config:/config \
     -v ./data:/data \
     ghcr.io/chattocorp/chatto:latest \
     init -c /config/chatto.toml
   ```

   The Docker image's entrypoint runs the bundled `chatto` binary, so `init -c /config/chatto.toml` is the container equivalent of running `chatto init -c /config/chatto.toml`. This writes `chatto.toml` with generated secrets into `config/`. In the Docker image, the generated embedded NATS data directory resolves to the `/data` mount.

   `PUID` and `PGID` keep generated files owned by your host user when bind-mounting local directories. If you omit them, the image defaults to `1000:1000`.

   The config file is optional. Every `chatto.toml` setting can also be provided through environment variables, which is how the Docker Compose guide configures Chatto.

3. **Review the generated config:**

   Open `config/chatto.toml` in your editor and review the generated values before starting Chatto. The local Quick Start defaults work as-is, but this is where you can change settings such as `webserver.url`, owner emails, SMTP, storage, or login providers.

4. **Start the server:**

   ```bash
   docker run --name chatto --rm -p 4000:4000 \
     -e PUID="$(id -u)" \
     -e PGID="$(id -g)" \
     -v ./config:/config \
     -v ./data:/data \
     ghcr.io/chattocorp/chatto:latest
   ```

5. **Open your browser** to [http://localhost:4000](http://localhost:4000).

</Steps>

## From Binary

<Steps>

1. **Install Chatto:**

   ```bash
   brew install chattocorp/tap/chatto
   ```

   You can also download the binary manually from [GitHub Releases](https://github.com/chattocorp/chatto/releases).

2. **Generate a configuration file:**

   ```bash
   chatto init
   ```

   This creates a `chatto.toml` with random secrets for cookie signing, cookie encryption, runtime-token verifiers, asset signing, and NATS authentication.

3. **Start the server:**

   ```bash
   chatto run
   ```

4. **Open your browser** to [http://localhost:4000](http://localhost:4000).

</Steps>

The generated `chatto.toml` includes random secrets, embedded NATS storage, authentication defaults, and commented examples for external NATS and login providers. The embedded NATS TCP listener is commented out by default. Leave it that way for the simplest single-process setup; uncomment `nats.embedded.port` when you need CLI tools such as `chatto backup` or `chatto keys export` to connect to the running embedded NATS server.
